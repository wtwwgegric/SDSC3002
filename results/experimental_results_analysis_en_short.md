# Experimental Results and Analysis

We compared three item-similarity retrieval strategies under a binary-preference Jaccard setting: naive brute-force search, an optimized sparse exact baseline, and an approximate MinHash+LSH pipeline. On MovieLens 1M, the results show a clear runtime crossover. The naive baseline becomes inefficient quickly because it scales quadratically with the number of items, while the sparse exact baseline remains competitive by exploiting sparsity. However, once the subset size grows to roughly 900-1000 items, LSH begins to outperform the sparse exact baseline, and from 1200 items onward the advantage becomes stable. At 3000 items, LSH reduces runtime to 0.975 seconds, compared with 2.099 seconds for sparse exact and 6.491 seconds for naive brute-force, while achieving recall@10 of 0.565. This indicates that on moderately large datasets, LSH can provide a meaningful speed-accuracy tradeoff.

The Amazon Musical Instruments 5-core dataset leads to a different conclusion. Because the data are much sparser, the sparse exact baseline is exceptionally strong: only a small fraction of all theoretical item pairs have nonzero overlap, so exact computation remains efficient even at full scale. Under the same LSH configuration, no runtime crossover appears on Amazon. On the full 10,606-item dataset, sparse exact finishes in 0.665 seconds, whereas LSH requires 1.814 seconds and reaches recall@10 of only 0.272. Additional tuning shows that recall can be increased substantially by using more aggressive banding, especially with one row per band, but this comes with a large runtime penalty. Therefore, the main conclusion is that LSH is not universally superior to exact search; its practical value depends strongly on dataset sparsity and on how efficient the exact baseline already is.
